# QA Playbook

Этот документ описывает, как организовать ручную команду тестировщиков вокруг NewDiary без хаоса и разночтений.

## 1. Окружения

| Окружение | Для чего | Кто использует |
| --- | --- | --- |
| Локальное dev | Быстрая проверка отдельных PR и UI-состояний | разработчики |
| Серверное test/prod-like | Основной smoke и ручная регрессия после выката | QA, релиз-ответственный |
| Storybook artifact | Визуальная проверка компонентов и состояний без runtime | разработчики, дизайнер, QA |

## 2. Набор ролей для тестирования

QA-команде нужен устойчивый комплект пользователей:

- `admin`
- `organizer`
- `curator`
- `participant`

Рекомендуемый подход:

- постоянные тестовые аккаунты для smoke;
- отдельные временные invite/login magic links для UAT и регрессии;
- хранение матрицы аккаунтов в защищённом внутреннем документе, а не в публичном репозитории.

## 3. Ежедневный ритм команды

### После каждого server deploy

1. Релиз-ответственный сообщает image tag и среду.
2. QA проходит [Smoke checklist](/qa/smoke-checklist).
3. Все найденные дефекты фиксируются по шаблону из [Bug report template](/qa/bug-report-template).
4. После фикса разработчик сообщает tag/commit для ретеста.

### Перед пользовательской демонстрацией

1. Пройти smoke повторно.
2. Проверить ключевые роли на мобильном устройстве.
3. Убедиться, что magic links и setup/admin не сломаны.

## 4. Regression matrix

Минимальный регрессионный набор:

### Public / auth

- регистрация участника на публичный заезд;
- setup первого администратора;
- login magic link;
- invite magic link для участника и куратора.

### Organizer

- открыть workspace заезда;
- проверить программу, дни, события;
- добавить event-level вопрос рефлексии, day-level вопрос рефлексии и проверить сохранение флага обязательности;
- publish/draft flow;
- группы, участники и настройки заезда.

### Participant

- открыть `Состояние`;
- выбрать день;
- отметить состояние по событию;
- ответить на настроенные вопросы мероприятия или оставить legacy-комментарий, если вопросов нет;
- проверить блокировку перехода при пустом обязательном вопросе мероприятия;
- пройти дневную рефлексию и проверить, что обязательный вопрос дня влияет на завершённость прогресса;
- проверить mobile layout.

### Curator

- открыть `Пульс дня`;
- проверить empty/ready state;
- убедиться, что group-scoped данные не смешиваются;
- проверить мобильный layout и интерактивные графики.

### Admin

- загрузить workspace;
- просмотреть пользователей;
- назначить пользователя на session/group;
- сгенерировать login magic link.

## 5. Что тестируем только на mobile

- participant daily flow как основной пользовательский сценарий;
- sticky headers / accordions / day switcher;
- curator dashboard на узком экране;
- отсутствие общего горизонтального overflow.

## 6. Правила ретеста

- дефект закрывается только после ретеста на той же роли и в той же среде;
- если баг был device-specific, ретест обязателен на том же типе устройства;
- если баг затронул deploy/runtime, ретест идёт после нового server tag, а не после локальной проверки.
