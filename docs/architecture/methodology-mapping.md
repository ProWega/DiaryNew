# Methodology mapping: «Дневник пути»

Этот документ — терминологический и архитектурный контракт между методикой Истоков «Дневник пути» (исследование 148 интервью, 2025) и фактическим кодом NewDiary. Он не описывает «фичи»: он показывает, **где код уже соответствует методике, где расходится, и где из-за расхождения может быть тихо нарушено правило «что нельзя нарушать»**.

Документ — обязательное чтение перед любым PR, который трогает дневник участника, рефлексию или дашборд куратора.

## Зачем

NewDiary исторически построен как **универсальная role-based платформа дневника состояния**. Методика «Дневник пути» — не очередная фича: это переопределение продукта на специфический методологический контур с жёсткими границами. Шкалы, агрегаты, формулировки и сами артефакты, которые видит куратор, — другие. Без этого mapping'а часть требований методики неизбежно нарушится в одном из 30+ мест, где сейчас код агрегирует состояния или показывает «риск».

## 1. Что нельзя нарушать (technical contract)

Жёсткие правила из секции VIII методики, переписанные как **технические инварианты с указанием места их защиты**.

| #   | Правило                                      | Где и как защищается                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| --- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Ненавязчивость**: все поля диария optional | [server/validation/schemas.cjs](../../server/validation/schemas.cjs) — `updateDiaryEntrySchema`, `updateReflectionSchema`. Сейчас единственное required поле — `dayId` (нужен для роутинга), это ОК. **Запрещено** добавлять `min(1)` к `comment`, `mind`, `heart`, `will`, `freeText`                                                                                                                                                                                                                                                                                                   |
| 2   | **Никаких «диагнозов»**: Сбой ≠ риск-метка   | Удалить из view: `RiskScatterChart`, `EventImpactBarChart` (события «с риском»), `buildCuratorRiskEventRows`. Запретить endpoint'ы вида `/risk/<userId>` или `/burnout-flags`                                                                                                                                                                                                                                                                                                                                                                                                            |
| 3   | **Авторство**: запись принадлежит участнику  | Колонки `is_anonymous`, `is_hidden_from_curator` в `diary_entries` и `daily_reflections` + **центральная фильтрация в [server/db/repositories/diaryStore.cjs](../../server/db/repositories/diaryStore.cjs)** (одно место). Любой curator-read проходит через этот фильтр                                                                                                                                                                                                                                                                                                                 |
| 4   | **Никакой арифметики на состояниях**         | Удалить `averageActivation`, `buildCuratorDistributionRows`, `buildCuratorScatterData` из [src/views/CuratorDashboard/CuratorDashboardView.jsx](../../src/views/CuratorDashboard/CuratorDashboardView.jsx). Удалить `score` и `averageActivation` из ответа `/curator/.../dashboard`                                                                                                                                                                                                                                                                                                     |
| 5   | **Тишина — равноправный знак, не дно**       | В `STATE_SCALE_META` — Тишина не должна иметь визуально «худший» цвет (нынешняя `panic` красная — это OK для Сбоя, но Тишина не должна быть «бледной»/«слабой»). В сортировке списков — Тишина не последняя                                                                                                                                                                                                                                                                                                                                                                              |
| 6   | **Подростки и Тишина — мягкие пороги**       | Знак «нужен разговор» для участника в Тишине срабатывает не на 3 Сбоя подряд, а на 2 (или вовсе не срабатывает — ответственность куратора). Подросток (12–16) — отдельный feature flag, не упрощённая копия взрослого                                                                                                                                                                                                                                                                                                                                                                    |
| 7   | **Кнопки приватности per-record**            | Не «один глобальный анонимный режим», а **на каждой записи** два независимых **чекбокса**: «анонимно для куратора» и «скрыть от куратора». UX-решение: именно checkboxes, не радиогруппа из 3 уровней — два чекбокса прямо отражают два булевых флага в БД и не вводят искусственных ограничений на их комбинации. Защищается на уровне UI ([PrivacyToggle.jsx](../../src/components/methodology/PrivacyToggle.jsx)) и на уровне API ([server/lib/privacy.cjs](../../server/lib/privacy.cjs) — central filter: `is_anonymous` обезличивает, `is_hidden_from_curator` полностью скрывает) |

**Тесты, которые блокируют регрессию:** см. секцию 5.

## 1.5. Глоссарий слов (technical contract)

Из методического приложения «На чём это стоит». Это **inviolable правило**: каждый текст в UI, каждое название endpoint, каждая copy-строка проверяется по этим спискам.

**Работают в UI (allow-list):** путь, тетрадь, запись, заметка, отозвалось, подвинулся, понимаю, целостность, собирание, лад, тишина, настройка, подъём, сбой, рядом, в стороне, опора, передача, поиск, проверка, бережно, бережение, свои, спутник, дорога.

**НЕ работают (ban-list):** уровень, стадия, прогресс, стрик, стрикинг, метрика, оценка, диагноз, скилл, статус, ментор, эксперт, зрелость (как этап), коммьюнити, миссия (в маркетинговом смысле), холизм, харизма, рост (как достижение), провал, регресс.

**Работают только как ТОН (фон, не на поверхности интерфейса):** подвиг, послушание, покаяние, благодать, благоговение, кенозис, синергия, соборность, исихия. Эти слова — мощные методологические опоры, но в светской программе должны звучать как воздух, не как термины.

**Принцип проверки** (по А.А. Ухтомскому): каждое UX-решение проверяется вопросом «строит ли это доминанту на участника или на нас?». Если на нас — переделать.

## 2. Mapping таблиц

### 2.1. Шкала состояний

**Текущая** ([src/data/stateScaleModel.js](../../src/data/stateScaleModel.js), `STATE_SCALE_ORDER` строки 24–32): **7 уровней с 3 зонами**:

```
apathy / passive / relaxed / balance / engaged / overstimulated / panic
zones: burnout / integration / distress
```

**Методика** (раздел III, шкала ν5): **5 уровней без зон**:

```
Тишина / Настройка / Лад / Подъём / Сбой
```

#### Решение: вариант B (переразметка 7 → 5)

Старый `state_id` остаётся для совместимости, добавляется `methodology_label` в `STATE_SCALE_META`. UI показывает методические 5 уровней, БД хранит 7. История сохраняется, миграции БД не требуется.

**Финальный mapping 7 → 5:**

| Текущий ID           | →   | Методика  | Цвет (текущий → методический) |
| -------------------- | --- | --------- | ----------------------------- |
| `apathy` (0)         | →   | Тишина    | глубокий синий — оставить     |
| `passive` (1)        | →   | Тишина    | глубокий синий                |
| `relaxed` (2)        | →   | Настройка | серебристо-серый              |
| `balance` (3)        | →   | Лад       | тёплый зелёный                |
| `engaged` (4)        | →   | Подъём    | янтарный                      |
| `overstimulated` (5) | →   | Сбой      | алый                          |
| `panic` (6)          | →   | Сбой      | алый                          |

`apathy/passive` обе → Тишина (нижняя группа), `overstimulated/panic` обе → Сбой (верхняя группа). Внутри каждой группы сохраняется текущий `state_level` для гранулярности — методический UI группирует, БД хранит точно.

### 2.2. Reflection (итог дня)

| Текущее                                                                                                                     | Методика                                                                                 |
| --------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| `daily_reflections.answers` (jsonb) с ключами `q1 / q2 / q3 / freeText` ([schema.sql:191–201](../../server/sql/schema.sql)) | Структурно: `mind` / `heart` / `will` + `free_text` + опциональное «главное событие дня» |
| `program_days.reflection_prompts` (jsonb) — массив строк ([schema.sql:132–145](../../server/sql/schema.sql))                | `mind`/`heart`/`will` промпты + меняются под наcтрой участника                           |
| Один универсальный текст вопросов                                                                                           | 4 набора вопросов, по числу настроев                                                     |

**Решение**: `daily_reflections.answers.jsonb` гибкий — можно хранить новые ключи без миграции. Меняем zod-контракт `updateReflectionSchema` (добавить `mind / heart / will` без удаления старых `q1/q2/q3` — старый клиент продолжает работать в переходный период).

**Маппинг для legacy данных** (open decision #5):

- Вариант 1: оставить как есть, при чтении в новом UI показывать «архивная запись»
- Вариант 2: автоматический backfill `q1 → mind`, `q2 → heart`, `q3 → will` (риск — семантически вопросы могли быть другими)

### 2.3. Diary entry (отметка возле события)

**Текущее** ([schema.sql:174–189](../../server/sql/schema.sql)):

```
diary_entries: state_id, state_level, comment, confidence, meta jsonb, responded_at, ...
```

**Что добавить:**

| Поле                     | Тип                                            | Где хранить                                                                      | Зачем                                                                 |
| ------------------------ | ---------------------------------------------- | -------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| `group_lad`              | text (one of `with_group / alongside / apart`) | `meta jsonb` (без миграции) ИЛИ отдельная колонка (для индексирования куратором) | Лад с группой — второе измерение состояния (методика, секция III.2.2) |
| `is_anonymous`           | bool default false                             | Отдельная колонка (фильтрация в SQL)                                             | Per-record анонимность (правило 3)                                    |
| `is_hidden_from_curator` | bool default false                             | Отдельная колонка (фильтрация в SQL)                                             | Полное скрытие от куратора                                            |

**Privacy-флаги — не в meta**: они должны участвовать в SQL `WHERE` и фильтре чтения. JSONB-колонка для этого неудобна.

### 2.4. Этап пути (journey_stage) + флаг «бережно» — НОВЫЕ сущности

Семантически: **этап пути ≠ настрой**. Настрой — настроение/намерение «на смену». Этап пути — «где я в течение жизни». Этапы циклически переживаются заново — педагог в Передаче может на новой смене оказаться в Поиске. Это не регресс, а живое движение.

«Тишина» **не один из этапов**. Она стала отдельным флагом `is_careful_mode`, который ставится поверх любого этапа. Обоснование v4 §I.5: острая нагрузка (СВО, болезнь близкого, особый ребёнок) почти всегда сопрягается с другим этапом — мама особого ребёнка одновременно в Передаче и в нагрузке. Заставлять выбирать = терять правду.

| Где                     | Что добавить                                                                                                                                |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Таблица `session_users` | Колонки `journey_stage text` (`search / verification / support / transmission`) + `is_careful_mode boolean default false`                   |
| API                     | `PATCH /api/participant/sessions/:sessionId/journey-stage` принимает `journeyStage` + `isCarefulMode` (оба optional)                        |
| Audit log               | Смена этапа / переключение carefulMode — события через [server/services/auditLog.cjs](../../server/services/auditLog.cjs)                   |
| UI                      | Onboarding-screen `/onboarding/journey-stage` после первого входа в дневник; `CarefulModeToggle` доступен на странице как отдельный чекбокс |

Тон вопросов рефлексии меняется по этапу (`REFLECTION_PROMPTS_BY_STAGE` в `src/data/methodology.ts`). Когда `is_careful_mode=true` поверх любого этапа применяются мягкие промпты `REFLECTION_PROMPTS_CAREFUL` (без императивов «опишите/оцените/обязательно»).

### 2.5. Curator dashboard — РАДИКАЛЬНЫЙ передел

## 3. Фазы миграции

### Фаза 1 — Constants & types (1–2 дня, риск нулевой)

- Создать `src/data/methodology.ts` — константы `MOOD`, `GROUP_LAD`, `SUMMARY_AXES` + методические `STATE_LABELS` (5) с mapping-таблицей на текущие 7 id'ов
- Расширить [src/types/domain.ts](../../src/types/domain.ts) — `Mood`, `GroupLad`, `SummaryAxis`, `StateLabel`
- Vitest: тест на согласованность `STATE_SCALE_ORDER` ↔ методический `STATE_LABELS`
- Никаких изменений UI / БД. Чистая constants-доступность

**Граница**: после фазы 1 нигде в коде не должно остаться захардкоженных строк типа `"balance"` / `"баланс"` — только импорты из methodology.ts.

### Фаза 2 — Schema (расширение, не замена) (1 день)

- Миграция через `node-pg-migrate`:
  ```sql
  ALTER TABLE diary_entries
    ADD COLUMN is_anonymous bool not null default false,
    ADD COLUMN is_hidden_from_curator bool not null default false,
    ADD COLUMN group_lad text;
  ALTER TABLE daily_reflections
    ADD COLUMN is_anonymous bool not null default false,
    ADD COLUMN is_hidden_from_curator bool not null default false;
  ALTER TABLE participant_diaries
    ADD COLUMN mood text;
  ```
- Старые колонки нетронуты. Default-значения такие, что старый код продолжает работать.
- Repository-слой [server/db/repositories/diaryStore.cjs](../../server/db/repositories/diaryStore.cjs): добавить **центральный фильтр приватности** для curator read-path (одно место, тестом покрытое).

**Граница**: после фазы 2 БД готова, но новые поля никем не используются. Прод-деплой в этой точке безопасен.

### Фаза 3 — Frontend поверх новых полей (3–5 дней)

- StateScalePicker: подменить `STATE_SCALE_META` (вариант B) — мап labels/colors на 5 методических групп
- Reflection editor: три textarea Ум/Сердце/Воля + free-text. Старые ключи `q1/q2/q3` маппятся на новые в адаптере на бэке
- Mood selection screen — новый route `/onboarding/mood`, появляется при первом входе в диарий смены
- Per-record privacy toggles (две checkbox) рядом с textarea
- Group-lad picker рядом со StateScalePicker (опционально — пропускается без напоминаний)

**Граница**: после фазы 3 участник видит методический интерфейс, но куратор всё ещё видит старый дашборд.

### Фаза 4 — Curator radical redesign (5–10 дней, отдельный PR)

- Новый view `src/views/CuratorBrief/` с тремя секциями: записка / карточка / партитура
- Старый CuratorDashboard под feature flag `legacy_curator_dashboard` — **пока не удалять**, держать 1–2 цикла обратной связи
- Бэк: `narrativeBriefService.cjs` + новый endpoint `GET /api/curator/sessions/:sessionId/groups/:groupId/brief`
- Удалить из ответа `/curator/.../dashboard` все агрегаты — после миграции куратора на новый view
- Удалить chart-компоненты (`RiskScatterChart`, `StackedDistributionChart`, `EventImpactBarChart`, `GroupScore`)

**Граница**: после фазы 4 продукт методически целен.

### Фаза 5 — Жизнь после смены (отдельный план, ~2 недели)

- Cron / scheduler (BullMQ или временной)
- Touchpoint-таблица + email-уведомления
- Карта своих + opt-in flow

Не блокирует фазы 1–4. Можно отложить на следующий квартал.

## 4. Decisions

Решения команды (зафиксированы, можно стартовать Фазу 1):

| #   | Вопрос                  | Решение                                                                                                                                                                                                              |
| --- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Шкала                   | **Вариант B** — переразметить 7 → 5 групп. UI показывает методические 5, БД хранит 7. История сохраняется. Mapping для `passive` и `overstimulated/panic` — см. таблицу в 2.1, точки 2 и 6 уточнить в фазе 1         |
| 2   | Подростки 12–16         | **Phase 5+**. Отдельный UX-трек после стабилизации основного контура. В фазах 1–4 поле возраста просто фиксируется — никакого переключения интерфейса                                                                |
| 3   | AI в narrativeBrief     | **Правила в фазе 4, LLM в отдельной задаче.** В фазе 4 `narrativeBriefService.cjs` строится на детерминистических правилах. LLM-вариант — следующий квартал, отдельный план с промпт-инжинирингом и оценкой качества |
| 4   | «Карта своих»           | **Opt-in уровня смены.** Один чекбокс при регистрации/итоге смены — «делиться контактом со своей группой». Per-record opt-in не делаем                                                                               |
| 5   | Backfill старых записей | **Legacy view без backfill.** Старые `q1/q2/q3` отображаются «архивной» вёрсткой. Mapping `q1→mind` семантически не гарантирован, не делаем автоматически                                                            |
